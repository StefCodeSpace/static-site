Reference · June 2026

D365 Success by Design — Taxonomy

Go to SbD Delivery Framework

Key terms, work item types, hierarchy levels, SbD phases, and legacy equivalents across Sure Step, SAP/ASAP, Agile, SAFe, and traditional PMBOK frameworks.

Section 01
Framework Overview
Success by Design (SbD) is Microsoft's implementation framework for Dynamics 365, replacing Sure Step. It organises delivery around four conceptual "trees" in Azure DevOps, each answering a different question about the project.
Why this matters for DAC: SbD borrows vocabulary from Agile, SAFe, traditional ERP (ASAP/Sure Step), APQC, and PMBOK — but renames many concepts. Understanding the lineage avoids confusion when bridging the ADO BPC template with DAC's own PMO governance language.
WHY Tree

Business Process Catalog (BPC)

The why — a hierarchical, system-agnostic catalogue of every business process the organisation runs. Microsoft-published and versioned. The strategic anchor for all traceability.

WHAT Tree

Deliverables Catalogue

The what — work products required to realise the BPC scenarios: Configurations, Enhancements, Integrations, Migrations, Documentation, Security, Reports, and more.

GOALS Tree

Objectives

Goals set for a Sprint or Tollgate review, owned by functional workstreams. Structured under audience tier, functional area, and cadence-level (Project / Tollgate / Sprint) folders.

HOW / WHEN Tree

SbD Delivery Plan

The timeline backbone — Phases, Workshops, Tasks, and Documentation aligned to the SbD lifecycle phases (Strategize → Initiate → Implement → Prepare → Operate).

ADO Structure

Azure DevOps as Backbone

All four trees live in ADO as a single portfolio backlog. The Tree WIT delineates each tree's root. Area paths are shared — tree separation is by parent WIT title, not area path.

Traceability

Full Chain: Why → What → Test

Objective → BPC Scenario → SOP Documentation → Test Case. Bugs link back to BPC Scenarios via Related Links. This is the long-term strategic value of ADO in the SbD stack.

Core Vocabulary at a Glance

Fit-to-Standard

SbD's guiding philosophy: default to standard D365 processes and adapt the business, rather than customising the system. This is a design principle, not a replacement for Fit/Gap analysis — SbD still performs Fit/Gap (via the Fit Gap Status field on every Scenario). The two coexist: your starting bias is "fit to standard first."

Scenario Board

A visual facilitation tool used in discovery workshops to map the end-to-end process with stakeholders. Equivalent to a JAD session output or high-level use-case diagram.

Tollgate

A go/no-go review checkpoint at the end of an Iteration (delivery period). It is the gate event, not the time container — the Iteration holds the Sprints, the Tollgate reviews outcomes against Objectives. ≈ Stage Gate (PMBOK) / Phase Gate (Sure Step). DAC's PMO calls this "Stage Gate."

Storyline / Happy Path

The primary end-to-end scenario walkthrough agreed in the Scenario Board workshop. Becomes the basis for the Storyline Design Review workshop (fit-to-standard demo).

RAID (Distributed)

SbD doesn't use a single RAID log. Instead, Risks, Issues, Assumptions, Decisions, and Change Requests are individual ADO work item types with their own fields and backlogs.

APQC Mapping

Each BPC Process item carries optional APQC ID and APQC Description fields, anchoring the process to the industry-standard American Productivity & Quality Center framework.

Section 04
SbD Lifecycle Phases
Success by Design defines five phases. Each maps to antecedent phases from Microsoft Sure Step and traditional PMBOK, and to SAFe cadence constructs. The "Partner phase" field on BPC items lets you map your own phases to the SbD standard.
01
Strategize
Sure Step: Diagnostic · SAFe: PI Planning
Vision, transformation roadmap, change management strategy, organisational readiness, high-level scope confirmation.
02
Initiate
Sure Step: Analysis · PMBOK: Planning Gate
Project mobilisation, governance setup, initial BPC scoping, environment provisioning, team onboarding.
03
Implement
Sure Step: Design + Development · Agile: Sprints
Iterative fit-to-standard workshops → configuration → enhancement → integration → migration development. Core delivery phase, runs in an iterative cadence of Sprints grouped into Iterations, each closing with a Tollgate review.
04
Prepare
Sure Step: Deployment · SAP: ASAP Preparation
UAT (User Acceptance Testing), cutover planning, data migration runs, training, go-live readiness review, hypercare setup.
05
Operate
Sure Step: Operation · PMBOK: Closure Gate
Post-go-live hypercare, support transition, lessons learned, continuous improvement, project closure.

Cadence Constructs Within Implement

Tollgate

Tollgate — Review Checkpoint

A go/no-go review event at the end of a delivery period. It is the checkpoint, not the time container. The cadence period (the group of Sprints leading up to it) is the iteration. ≈ Stage Gate (PMBOK) / Phase Gate (Sure Step) / PI Review event (SAFe). Note: DAC's PMO calls this construct "Stage Gate".

Iteration

Iteration — Time Container

The delivery period that leads up to a Tollgate review. Contains multiple Sprints. In the ADO iteration path hierarchy, this is the parent level (e.g. "Tollgate 1") that holds Sprint children. ≈ Program Increment / PI (SAFe) / Release (Scrum). The distinction matters: the Tollgate is the gate at the end; the Iteration is the block of time before it.

Sprint

Sprint — Time-box

Standard two-week Agile time-box within an Iteration. Carries sprint-level start/end dates in the ADO iteration path. ≈ Scrum Sprint / Agile Iteration.

Workshop Pattern

Scenario Board → Storyline Review

Two-workshop pattern per end-to-end:
(1) Scenario Board Discovery — scope and happy path alignment.
(2) Storyline Design Review — fit-to-standard demo, 80–90% acceptance target. Detailed Design Workshops follow and produce the FDDs.

SbD Phase Field

SbD Phase on BPC Items

Each BPC Scenario carries a "Success by Design phase" field indicating which implementation phase it should be delivered in. Defaults to Undetermined in the source BPC data — requires manual population during delivery planning or sprint assignment. Not auto-populated by Microsoft.
